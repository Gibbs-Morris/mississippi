# RFC 2119 — Key Words for Indicating Requirement Levels

Technical specifications live or die by their precision. A single ambiguous
word — "should", "may", "must" — can mean entirely different things to different
readers, leading to incompatible implementations, unenforceable contracts, and
silent non-compliance. RFC 2119 eliminates that ambiguity by assigning exact,
universally agreed meanings to ten keywords used in standards and specification
documents.

**This document is written in Minto Pyramid format.**

---

## Governing Thought

RFC 2119, authored by Scott Bradner and published by the IETF in March 1997,
defines ten keywords — MUST, MUST NOT, REQUIRED, SHALL, SHALL NOT, SHOULD,
SHOULD NOT, RECOMMENDED, MAY, and OPTIONAL — that convey precise obligation
levels in technical specifications. When used consistently and correctly, these
keywords transform prose requirements into unambiguous, testable contracts that
implementers, reviewers, and automated tools can interpret identically.

---

## Situation

Standards bodies, open-source projects, and engineering teams write
specification documents to communicate how systems should behave. These
documents contain hundreds of individual requirements, each carrying a different
level of obligation. Some requirements are absolute — violating them means the
implementation is non-conformant. Others are strong recommendations that allow
exceptions in justified circumstances. Still others are genuinely optional
features that implementations may or may not include.

## Complication

Natural language is inherently ambiguous. The word "should" in everyday English
can mean anything from a moral imperative to a casual suggestion. When a
specification says "the server should validate the token", readers cannot tell
whether validation is an absolute requirement, a strong recommendation, or
merely a nice-to-have. This ambiguity produces divergent implementations:
one team treats the requirement as mandatory and rejects invalid tokens; another
treats it as advisory and silently accepts them. Interoperability breaks, and
the specification has failed at its primary purpose.

## Question

How can specification authors convey precise, unambiguous obligation levels so
that every reader — human or machine — interprets each requirement identically?

---

## Key-Line 1: The Ten Keywords Have Precise, Non-Overlapping Meanings

RFC 2119 defines exactly ten keywords, organised into three obligation tiers.
Each keyword has one and only one meaning, and the tiers do not overlap.

### Absolute Requirements (No Exceptions)

| Keyword | Meaning |
|---------|---------|
| **MUST** | The requirement is an absolute obligation. An implementation that fails to satisfy a MUST requirement is non-conformant. There are no circumstances under which violation is acceptable. |
| **REQUIRED** | Synonym for MUST. Identical semantics; used where it reads more naturally in a sentence. |
| **SHALL** | Synonym for MUST. Identical semantics; common in legal and formal standards traditions. |
| **MUST NOT** | An absolute prohibition. The described behaviour is forbidden under all circumstances. An implementation that exhibits a MUST NOT behaviour is non-conformant. |
| **SHALL NOT** | Synonym for MUST NOT. Identical semantics. |

These five keywords represent the ceiling and floor of conformance. They
define what an implementation absolutely must do and absolutely must not do.
There is no room for judgement, context, or trade-off.

### Strong Recommendations (Exceptions Require Justification)

| Keyword | Meaning |
|---------|---------|
| **SHOULD** | The requirement is a strong recommendation. There may exist valid reasons in particular circumstances to ignore it, but the full implications must be understood and carefully weighed before choosing a different course. |
| **SHOULD NOT** | The behaviour is strongly discouraged. There may exist valid reasons in particular circumstances to permit it, but the full implications must be understood and the case carefully weighed. |
| **RECOMMENDED** | Synonym for SHOULD. Identical semantics. |

SHOULD and SHOULD NOT acknowledge that real-world implementations face
constraints — performance budgets, hardware limitations, backwards-compatibility
obligations — that may justify deviation. The critical requirement is that the
implementer must understand the trade-off and make a deliberate, documented
decision rather than ignoring the requirement through oversight or
carelessness.

### Truly Optional (Implementer's Discretion)

| Keyword | Meaning |
|---------|---------|
| **MAY** | The item is genuinely optional. An implementation may include the feature or omit it entirely, and both choices are fully conformant. |
| **OPTIONAL** | Synonym for MAY. Identical semantics. |

MAY does not mean "unimportant". It means that the specification authors
considered the feature and deliberately decided that conformance does not
depend on it. Implementations that include a MAY feature must still implement
it correctly if they choose to include it.

## Key-Line 2: Correct Usage Requires Deliberate Practice

The keywords are powerful precisely because they are constrained. Misuse
weakens them and reintroduces the ambiguity they were designed to eliminate.

### Use Uppercase Only for Normative Force

RFC 2119 itself states: "These words MUST NOT be used in a manner that would
be considered purely informational." The convention is to use the keywords in
UPPERCASE when they carry normative meaning. When the same words appear in
lowercase ("the server must handle this case"), they carry no RFC 2119 force
and are ordinary English prose. This distinction is essential — it is the
mechanism by which readers and automated tools distinguish normative
requirements from explanatory text.

### One Requirement per Statement

Each normative statement should contain exactly one keyword and express exactly
one testable requirement. Compound statements ("the client MUST validate the
token and SHOULD cache the result") conflate an absolute requirement with a
recommendation, making conformance testing ambiguous. Split them into separate
statements.

### Provide Rationale for Strong Requirements

When a specification uses MUST or MUST NOT, readers benefit from understanding
why the requirement is absolute. A brief rationale — security, interoperability,
data integrity — helps implementers prioritise correctly and prevents the
perception that the keyword was chosen arbitrarily.

### Reserve MUST for True Invariants

Overusing MUST dilutes its force. If every recommendation in a specification is
marked MUST, implementers cannot distinguish genuine invariants from strong
preferences. Reserve MUST for requirements where violation would cause
non-interoperability, security vulnerabilities, data corruption, or protocol
failure. Use SHOULD for requirements that are important but where justified
exceptions exist.

### Never Use MUST When You Mean SHOULD

This is the single most common misuse. Authors frequently write MUST when they
know that real-world implementations will sometimes deviate for valid reasons.
The correct keyword is SHOULD — it communicates the strong recommendation while
honestly acknowledging that exceptions exist. Writing MUST when you mean SHOULD
creates specifications that are technically unimplementable, which trains
readers to ignore MUST requirements entirely.

## Key-Line 3: Interoperability Depends on the MAY/MUST Interaction

One of RFC 2119's most subtle and frequently misunderstood aspects is the
relationship between MAY features and the implementations that interact with
them.

When a specification says "an implementation MAY include feature X", it creates
an asymmetric obligation:

- **The implementer of feature X** has no obligation to include it but must
  implement it correctly if included.
- **Other implementations that interact with feature X** MUST be prepared to
  interoperate both with implementations that include the feature and with
  implementations that omit it.

This is the interoperability contract: optional features must not break
mandatory behaviour. If implementation A includes an optional extension, and
implementation B omits it, A must not rely on B supporting the extension, and B
must not fail when A uses it.

This principle extends to SHOULD as well. An implementation that omits a SHOULD
feature must still interoperate with implementations that include it, and vice
versa.

## Key-Line 4: The Boilerplate Clause Activates the Keywords

RFC 2119 is not automatically in effect. Specifications that use the keywords
MUST include a reference to RFC 2119, typically in a standardised boilerplate
paragraph near the beginning of the document:

> The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD",
> "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be
> interpreted as described in RFC 2119.

Without this boilerplate, the keywords are ordinary English words and carry no
standardised meaning. The boilerplate serves as a signal to readers that the
document uses precise requirement levels and that the uppercase keywords are
normative.

In repository-scoped instruction files and engineering guidelines, the
convention is typically lighter — a section heading such as "Rules (RFC 2119)"
signals that the following bullets use the keywords with their standardised
meanings.

## Key-Line 5: Common Mistakes and How to Avoid Them

Understanding the keywords is necessary but not sufficient. Authors must also
recognise the patterns of misuse that undermine specification quality.

### Mistake: Using MUST for Aspirational Requirements

If a requirement cannot be enforced or tested, MUST is the wrong keyword.
SHOULD communicates the intent without creating an unenforceable mandate.

**Wrong:** "Implementations MUST respond within 100 milliseconds."
(Network latency makes this unenforceable as an absolute requirement.)

**Right:** "Implementations SHOULD respond within 100 milliseconds. Response
times exceeding 500 milliseconds indicate a performance defect."

### Mistake: Omitting MUST NOT for Security Invariants

Security-critical prohibitions demand MUST NOT, not SHOULD NOT. Weakening a
security invariant to a recommendation invites non-compliance.

**Wrong:** "Servers SHOULD NOT accept expired tokens."

**Right:** "Servers MUST NOT accept expired tokens."

### Mistake: Treating MAY as Unimportant

MAY means the feature is optional, not that it is trivial. Specifications
sometimes relegate critical-but-optional features to MAY without explaining
the implications of omission.

**Better:** "Implementations MAY cache responses. Implementations that do not
cache MUST still handle cache-control headers from peers that do."

### Mistake: Using Keywords in Non-Normative Text

Introductions, examples, rationale sections, and appendices should not use
RFC 2119 keywords in their uppercase form. Doing so creates false requirements
that confuse both human readers and automated compliance tools.

### Mistake: Keyword Stacking

Statements like "implementations MUST SHOULD" or "this SHOULD NOT MUST" are
syntactically and semantically invalid. Each statement gets exactly one
obligation level.

## Key-Line 6: Application Beyond IETF Standards

Although RFC 2119 originated in the IETF for internet protocol specifications,
its adoption has spread far beyond that domain.

### API Documentation and OpenAPI Specifications

API contracts use MUST and MUST NOT to define mandatory request/response
behaviour, SHOULD for recommended practices (pagination, rate-limit headers),
and MAY for optional extensions (webhook support, experimental endpoints).

### Internal Engineering Guidelines

Teams adopt RFC 2119 keywords in coding standards, architecture decision
records, and operational runbooks. This transforms vague guidance ("we should
probably log errors") into precise, auditable policy ("Error responses MUST be
logged with correlation identifiers").

### Regulatory and Compliance Documents

ISO standards, PCI-DSS requirements, and GDPR technical guidance all use
RFC 2119 or equivalent keyword conventions to distinguish absolute mandates
from recommended controls.

### Repository Instruction Files

This repository uses RFC 2119 keywords throughout its instruction files under
a consolidated "Rules (RFC 2119)" heading per file. Each rule is a single
bullet with one keyword and one requirement, followed by a short rationale.
This convention keeps normative content scannable and enforceable.

## Summary

RFC 2119 solves a fundamental problem in technical communication: the
ambiguity of natural language when expressing obligation levels. Its ten
keywords — three tiers of absolute, recommended, and optional — provide a
shared vocabulary that specification authors, implementers, reviewers, and
automated tools all interpret identically.

The keywords work only when used with discipline:

- **MUST and MUST NOT** are reserved for invariants where violation means
  non-conformance.
- **SHOULD and SHOULD NOT** express strong recommendations that permit
  justified, documented exceptions.
- **MAY** marks genuinely optional features that must not break interoperability
  when present or absent.
- Keywords appear in **uppercase only** when carrying normative force.
- Each normative statement contains **exactly one keyword** and **one testable
  requirement**.
- The **boilerplate clause** activates the keywords; without it, they are
  ordinary English.

When applied consistently, RFC 2119 transforms prose specifications into
precise, testable, unambiguous contracts — the foundation on which
interoperable systems are built.
