## Findings

### Finding F1: There is no repo-wide host builder interface family yet (`IClientBuilder`, `IGatewayBuilder`, `IRuntimeBuilder` absent)
- Evidence:
  - `grep_search` for `IClientBuilder|IGatewayBuilder|IRuntimeBuilder|interface\s+I[A-Za-z]+Builder` in `src/**/*.cs` returned no matches.
- Second source:
  - Existing startup API uses extension registrations (`AddInletClient`, `AddInletServer`, `AddInletSilo`) in:
    - `src/Inlet.Client/InletClientRegistrations.cs`
    - `src/Inlet.Gateway/InletServerRegistrations.cs`
    - `src/Inlet.Runtime/InletSiloRegistrations.cs`
- Implication for plan:
  - Builder APIs are net-new public contracts and must be introduced deliberately with clear ownership and migration strategy.

### Finding F2: A localized builder precedent already exists (`InletBlazorSignalRBuilder`)
- Evidence:
  - `src/Inlet.Client/InletBlazorSignalRBuilder.cs` provides fluent configuration methods (`AddProjectionFetcher`, `ScanProjectionDtos`, `WithHubPath`, `WithRoutePrefix`) and a terminal `Build()`.
- Second source:
  - `src/Inlet.Client/InletBlazorRegistrations.cs` wires it via `AddInletBlazorSignalR(..., Action<InletBlazorSignalRBuilder>? configure)` and calls `builder.Build()`.
- Implication for plan:
  - We can reuse this pattern as a design baseline and decide whether to keep concrete-class builders, add interfaces, or use both.

### Finding F3: Current architecture is strongly registration-extension based, hierarchical, and composition-oriented
- Evidence:
  - `src/DomainModeling.Runtime/AggregateRegistrations.cs` exposes granular registration extension methods (`AddAggregateSupport`, `AddCommandHandler`, `AddEventType`, etc.).
  - `src/DomainModeling.Runtime/SagaRegistrations.cs` and `src/DomainModeling.Runtime/UxProjectionRegistrations.cs` layer additional feature registration.
- Second source:
  - `service-registration.instructions.md` requires hierarchical `Add{Feature}()` extension pattern and sync-only registration.
- Implication for plan:
  - Builders should likely orchestrate existing extension methods rather than bypass or duplicate registration internals.

### Finding F4: Domain-wide composition is already generated per host surface
- Evidence:
  - `src/Inlet.Client.Generators/DomainClientRegistrationGenerator.cs` emits `DomainFeatureRegistrations` with aggregate/saga/projection composition methods.
  - `src/Inlet.Runtime.Generators/DomainSiloRegistrationGenerator.cs` emits `DomainSiloRegistrations` with aggregate/saga/projection composition methods.
- Second source:
  - `src/Inlet.Gateway.Generators/DomainServerRegistrationGenerator.cs` emits `DomainServerRegistrations` for server mapping composition.
- Implication for plan:
  - Builder design must account for generator-produced registration entry points and avoid creating parallel disconnected composition trees.

### Finding F5: Reservoir already has explicit feature-state/reducer/effect registration primitives
- Evidence:
  - `src/Reservoir.Core/ReservoirRegistrations.cs` includes `AddFeatureState<TState>`, `AddReducer<...>`, `AddActionEffect<...>`, `AddReservoir()`.
- Second source:
  - `src/Reservoir.Abstractions/State/IFeatureState.cs` and `src/Reservoir.Abstractions/State/IFeatureStateRegistration.cs` confirm feature-state contract model.
- Implication for plan:
  - `IFeatureStateBuilder` should likely compose these primitives and enforce conventional ordering/requirements.

### Finding F6: Documentation IA is currently minimal; no existing “proposed/future design patterns” section exists
- Evidence:
  - `docs/Docusaurus/docs/index.md` indicates docs are being rebuilt and is sparse.
  - `docs/Docusaurus/docs/` currently has limited top-level sections (`client-state-management`, `contributing`, `spring-sample`, plus a few single pages).
- Second source:
  - `docs/Docusaurus/sidebars.ts` uses fully autogenerated sidebar from filesystem, so new folders/pages auto-appear.
- Implication for plan:
  - We can add a dedicated proposal area by folder structure alone, but we should choose naming and scope carefully to avoid confusion with live guidance.

### Finding F7: Contract layering and abstractions policy supports introducing lightweight public builder interfaces
- Evidence:
  - `abstractions-projects.instructions.md` requires contracts in `*.Abstractions` and implementations in main projects.
- Second source:
  - `framework-patterns.instructions.md` + `projects.instructions.md` reinforce contract/implementation separation and discoverable DI patterns.
- Implication for plan:
  - A two-project split per area (abstraction interface + implementation builder) is repo-consistent if these builders become public DX contracts.

### Finding F8: Registration behavior constraints will shape builder API design
- Evidence:
  - `service-registration.instructions.md` forbids async registration work and mandates options-oriented overloads.
- Second source:
  - `shared-policies.instructions.md` and `csharp.instructions.md` enforce DI and logging conventions, no service locator, internal-by-default implementation posture.
- Implication for plan:
  - Builders must remain configuration-orchestration only (sync), expose options-driven APIs, and avoid hiding anti-patterns.
